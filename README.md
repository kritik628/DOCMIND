# DocMind — RAG-based Document Research Assistant

DocMind is a Retrieval-Augmented Generation (RAG) system that lets you upload documents and ask questions about them, with answers grounded strictly in the document content instead of relying on an LLM's general (and potentially hallucinated) knowledge.

## How it works

1. **Ingestion (`/upload`)** — A document is read, split into ~200-word chunks, and each chunk is converted into a 3072-dimensional embedding vector using Google's Gemini embedding model. Chunks and their embeddings are stored persistently in a ChromaDB vector database.
2. **Retrieval (`/ask`)** — An incoming question is embedded the same way, and ChromaDB is queried to find the most semantically similar stored chunks (via cosine similarity), regardless of whether the question shares any keywords with the source text.
3. **Generation** — The retrieved chunks are combined and passed to the Gemini LLM along with strict instructions to answer only from that context, and to explicitly say "I don't know" if the answer isn't present — minimizing hallucination.
4. **API layer** — The entire pipeline is exposed as a FastAPI REST service with Pydantic-based request validation, structured logging, and automated tests.

## Architecture

```
Document → Chunking → Embedding (Gemini API) → ChromaDB (vector store)
                                                      ↑
Question → Embedding (Gemini API) → Similarity Search ┘
                                                      ↓
                                    Retrieved chunks → LLM (grounded prompt) → Answer
```

## Tech Stack

- **Language:** Python
- **LLM / Embeddings:** Google Gemini API (`gemini-2.5-flash-lite`, `gemini-embedding-001`)
- **Vector Database:** ChromaDB (persistent, local)
- **API Framework:** FastAPI + Uvicorn
- **Validation:** Pydantic
- **Testing:** pytest
- **Containerization:** Docker
- **Secrets Management:** python-dotenv

## API Endpoints

| Method | Endpoint | Description |
|---|---|---|
| GET | `/` | Health check |
| POST | `/upload` | Chunk, embed, and store a document (`{"filepath": "notes.txt"}`) |
| POST | `/ask` | Ask a question, get a RAG-grounded answer (`{"question": "..."}`) |
| GET | `/documents` | Returns the total number of chunks currently stored |

Interactive API docs are auto-generated by FastAPI at `/docs` once the server is running.

## Running Locally (without Docker)

1. Clone the repo and create a virtual environment:
   ```
   git clone https://github.com/kritik628/DOCMIND.git
   cd DOCMIND
   python -m venv venv
   venv\Scripts\activate   # Windows
   ```

2. Install dependencies:
   ```
   pip install -r requirements.txt
   ```

3. Create a `.env` file in the project root with your Gemini API key:
   ```
   GEMINI_API_KEY=your_key_here
   ```
   (Get a free key at https://aistudio.google.com/apikey)

4. Run the server:
   ```
   uvicorn app:app --reload
   ```

5. Visit `http://127.0.0.1:8000/docs` to try the API interactively.

## Running with Docker

The application is fully containerized — no local Python environment setup needed beyond Docker itself.

```
docker build -t docmind-app .
docker run -p 8000:8000 --env-file .env docmind-app
```

Then visit `http://127.0.0.1:8000/docs` as above. The container includes all dependencies pre-installed via `requirements.txt`, guaranteeing the exact same environment regardless of the host machine.

## Running Tests

```
pytest -v
```

Covers core chunking logic and defensive error handling (e.g. missing files).

## Key Design Decisions

- **Chunking (~200 words)** balances retrieval precision against token limits — smaller chunks pinpoint relevant sections instead of blending an entire document's meaning into one vector.
- **Defensive error handling** throughout — every method touching file content or LLM calls checks for failure states (missing files, failed API calls) rather than assuming success.
- **Centralized configuration** (`config.py`) avoids hardcoded model names and parameters scattered across the codebase.
- **Structured logging** instead of print statements, for timestamped, severity-tagged diagnostics.
- **Dockerized** for environment consistency — the app runs identically regardless of host machine setup.
